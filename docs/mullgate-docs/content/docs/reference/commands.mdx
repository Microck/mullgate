The documentation repeatedly anchors itself to the following CLI help contract:

- `mullgate --help`
- `mullgate setup --help`
- `mullgate start --help`
- `mullgate status --help`
- `mullgate doctor --help`
- `mullgate autostart --help`
- `mullgate relays --help`
- `mullgate recommend --help`
- `mullgate config --help`

## Core workflow commands

### `mullgate setup`

Used to create or update the base configuration, including account, proxy credentials, routes, and exposure-related settings.

### `mullgate start`

Used to render or launch the runtime environment.

### `mullgate status`

Used to inspect current runtime state.

### `mullgate doctor`

Used to diagnose runtime, routing, and hostname or exposure drift.

### `mullgate autostart`

Used on Linux to install, inspect, or remove the `systemd --user` service that starts `mullgate start` at login.

### `mullgate config`

Used to inspect and modify configuration surfaces.

Important subcommands referenced in the docs include:

- `mullgate exposure`
- `mullgate export`
- `mullgate regions`
- `mullgate hosts`
- `mullgate validate --refresh`
- `mullgate path`

`mullgate export` is the proxy-list generation surface. It supports ordered repeated selector batches such as `--country se --city got --count 1 --region europe --provider m247 --count 2`, guided `proxies.txt` creation via `--guided` with country and region pick-lists, preview mode via `--dry-run`, stdout delivery via `--stdout`, and safe file overwrite behavior with `--force`.

`mullgate regions` prints the curated export region groups that `export --region ...` accepts.

`mullgate relays` is the relay-inspection surface. `relays list` shows matching relay metadata, `relays probe` ranks selected candidates by `ping` latency, and `relays verify` checks whether one configured route still exits through Mullvad on the published proxy protocols.

`mullgate recommend` turns broad selector intent into exact relay-hostname recommendations. By default it stays advisory and previews the exact route Mullgate would use. With `--apply`, it pins the recommended relay hostname into saved config and refreshes the runtime artifacts.

## Invocation forms

Mullgate can be invoked in three documented ways:

1. installed `mullgate` command
2. packed release asset
3. source-checkout execution using `node dist/cli.js ...` or `pnpm exec tsx src/cli.ts ...`

## How the commands fit together

A practical workflow usually looks like this:

1. `mullgate setup`
2. `mullgate exposure`
3. `mullgate export`
4. `mullgate regions`
5. `mullgate relays list`
6. `mullgate relays probe`
7. `mullgate recommend`
8. `mullgate hosts`
9. `mullgate validate --refresh`
10. `mullgate start`
11. `mullgate status`
12. `mullgate doctor`

## Reference practice

When in doubt, prefer the live CLI help output as the source of truth for flags, defaults, and command wording.
